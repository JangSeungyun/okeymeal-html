# 04.Testing_QA (테스트 및 1차 기능 심사 준비 단계) 가이드

개발된 서비스가 1차 심사(기능 심사)를 통과할 수 있도록 구동 안정성과 과제 연계 기능(안심 식당 추천, 산책로 연계)의 완결성을 검증하는 단계입니다.

## 📄 작성 필수 문서 지정

1.  **`Test_Plan.md` (테스트 계획 및 시나리오)**
    *   과제 5번의 핵심 시나리오(기저질환자의 맞춤 식당 검색부터 식후 산책까지) 중심의 E2E 테스트 시나리오.
2.  **`QA_Report.md` (QA 결과 보고서)**
    *   테스트 시나리오 수행 결과(Pass/Fail) 및 잔여 버그 내역.

---

## 📐 문서별 맞춤 템플릿 규칙

### 1. `Test_Plan.md`
```markdown
## 1. 테스트 개요
- 테스트 목적 및 범위 (기능 심사 대응)

## 2. 시나리오: [시나리오명] (예: 당뇨 환자의 식당 탐색 및 산책 코스 추천)
| Step | Action | Expected Result |
| --- | --- | --- |
| 1 | 당뇨 프로필 선택 후 식당 검색 | 혈당 급상승 위험이 적은 식당 리스트 노출 |
| 2 | 메뉴판 이미지 업로드 (OCR) | 고당류 메뉴에 대한 시각적 경고 표시 |
| 3 | 식당 상세 페이지 진입 | 인근 걷기 좋은 산책 코스 연계 정보 노출 |
```

### 2. `QA_Report.md`
```markdown
## 총괄 테스트 결과
- 전체 테스트 케이스 수: N
- Pass: X, Fail: Y

## Fail 항목 및 조치 현황
- **[이슈명]**: 에러 상세 현상 -> **[조치 예정일/담당자]**
```

---

## 🎨 시각화 HTML 생성 규칙

QA/테스트 결과는 가시성이 가장 중요하므로 다음 시각화 규칙을 따릅니다.

1. **차트 시각화**: 전체 Pass/Fail 비율을 Chart.js(도넛 차트 등)를 이용하여 페이지 상단에 렌더링.
2. **상태 기반 컬러링**: 표에서 Pass는 `text-green-600`, Fail은 `text-red-600` 등 색상을 직관적으로 부여.
3. **칸반 보드 뷰 (선택)**: Fail 항목 조치 현황은 'To Do', 'In Progress', 'Done' 형태의 간단한 보드 UI로 시각화.
