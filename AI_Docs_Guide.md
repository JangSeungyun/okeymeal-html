# 🤖 AI 범용 마스터 문서 작성 가이드

이 문서는 Gemini, Claude, Codex 등 다양한 AI 에이전트가 이 프로젝트의 문서를 작성하거나 코드를 설계할 때 준수해야 할 지침을 제공합니다.

## 🎯 핵심 지침 (Core Mandates)

1.  **일관성**: 모든 문서는 최상단 메타데이터(YAML)와 최하단 변경 이력을 반드시 포함해야 합니다.
    *   **문서 상태 추적(Status)**: YAML 메타데이터에 반드시 `status: [Draft | Review | Approved | Final]` 항목을 포함하여 현재 문서의 완성도를 명시하십시오. (예: 초안 작성 시 `Draft`, 리뷰 요청 시 `Review`, 승인 시 `Approved`, 더 이상 변경 없는 확정본은 `Final`) — 폐기 문서는 `Archived`.
2.  **가이드 준수 최우선 원칙**: 문서를 작성하기 전, 반드시 대상 폴더에 있는 **`_Stage_Guide.md` 파일을 먼저 읽고 그 안에 명시된 맞춤형 템플릿 양식을 정확히 차용**하여 작성하십시오.
3.  **언어**: 요청 프롬프트가 영어이더라도, **최종 문서 출력은 반드시 한국어**로 작성하십시오.
4.  **무결성 (No Omissions)**: 기존 문서를 업데이트할 때, 기존의 유효한 내용을 절대로 임의로 생략하거나 `(기존 내용 유지...)`와 같은 자리 표시자로 대체하지 마십시오. 반드시 **전체 내용을 포함하여 병합(Merge)**된 최종 결과물을 출력해야 합니다. **내용을 간략화 하거나 요약하지 마십시오.** 또한, 번역을 요청할때도 **내용을 간략화 하거나 요약하지 마십시오.**
5.  **명확한 수정 의도**: 문서를 수정할 때, 무엇을 바꾸고 싶은지 명확한 의도를 전달해야 합니다. 변경 사항의 범위(부분 수정, 전체 교체 등)와 예상되는 영향을 함께 제시하십시오.

## 📂 문서 관리 추천 규칙 (Best Practices)

1.  **원자적 수정 (Atomic Edits)**: 한 번의 수정 시 하나의 목적(기능 추가, 오타 수정 등)에 집중하여 변경 이력을 명확히 관리하십시오.
2.  **버전 넘버링 규칙**:
    - `vX.Y.Z` 형식을 따르되, 단순 오타나 서식 수정은 `Z`(Patch), 내용 추가 및 보완은 `Y`(Minor), 전체 구조나 컨셉 변경은 `X`(Major) 단위를 올립니다.
3.  **시각화 활용**: 복잡한 프로세스, 아키텍처, 데이터 흐름은 반드시 **Mermaid 다이어그램**을 사용하여 시각화하십시오. (예: `UI_UX_Flow.md`의 상태도, `DB_Schema.md`의 ERD 등)
4.  **HTML 시각화 변환 및 폴더 구조**: 작성된 마크다운을 HTML로 변환할 때는 루트 디렉토리의 `HTML_Visualization_Guide.md` 공통 가이드와 각 폴더의 `_Stage_Guide.md` 하단에 명시된 맞춤 시각화 규칙을 적용해야 합니다. **반드시 해당 단계 폴더 내부에 `html/원본_문서명/` 하위 폴더를 생성**하고 그 안에 HTML 파일을 저장하여 마크다운 파일과 섞이지 않도록 하십시오.

## 🛠 마스터 프롬프트 (Master Prompt)

AI 에이전트를 소환하거나 새로운 대화를 시작할 때 아래 프롬프트를 복사하여 입력하십시오.

### 🇰🇷 한국어 버전

> 당신은 '2026 관광데이터 활용 공모전 (②-2 웹·앱 구현 부문)'의 전문 테크니컬 라이터이자 소프트웨어 아키텍트입니다. 이 프로젝트의 문서는 `documents/` 폴더 내에 마크다운으로 체계적으로 관리되고 있습니다. 
> 
> 문서를 작성하기 전, **해당 문서를 생성할 폴더 안에 있는 `_Stage_Guide.md`를 먼저 읽고 지정된 템플릿 규칙을 반드시 준수**하십시오. 기존 내용을 누락하지 말고 전체 문서를 완전한 상태로 유지하며 작성하십시오. 모든 답변은 한국어로 작성하며, 문서의 최상단에는 메타데이터를, 최하단에는 변경 이력 표를 포함하십시오.
